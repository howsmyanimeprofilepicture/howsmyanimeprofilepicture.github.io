3.3. git restore / git checkout --

파일을 커밋한 "직"후,

약간 수정을 한후 터미널에 $git status를 입력하면

  • Changes not staged for commit라고 뜬다.

  • xxxxxxxxxx
    On branch master
    Changes not staged for commit:
        (use "git add <file>..." to update what will be committed)
        (use "git checkout -- <file>..." to discard changes in working directory)
        
            modified: 커밋전에수정안한텍스트파일임.txt

  • 변경사항이 스테이지에 올라오지 않았다는 이야기다.(변경사항이 Add되지 않았다는 이야기다.)

  • 해당 파일을 $git add를 통해 stage에 올리거나,

  • git checkout -- <file>...를 통해 (stage에 올라온 이후의) 수정사항을 도로 롤백시켜야 한다.

  • 현재 버전에서는 $ git restore <file>...로 롤백시키는 것 같다.

 

* 수정한 내용 복원하기 (git restore)

  • 마지막 커밋(HEAD)로 파일의 내용들을 롤백시키려면 $ git restore <file> ...를 사용하면 된다.
  • 만약에 해당 수정 내용이 (git add를 통해) 이미 stage에 넣어졌다 해도 git restore를 사용하면 HEAD 시점으로 파일의 내용을 롤백할 수 있다.
  • git restore 추가 이전에는, stage에 이미 넣어진 수정사항은 $ git reset HEAD <file> ...를 사용해서 수정사항을 unstage하고, 그다음 $ git checkout -- <file>...를 통해 롤백해야 했다고 한다.

4. Github과 git remote

 

xxxxxxxxxx
$ git remote add origin

 

  • $ git remote --help https://git-scm.com/docs/git-remote

  • git remote add [-t <branch>] [-m <master>] [-f] [--[no-]tags] [--mirror=(fetch|push)] <name> <url> []는 옵션이고 <>는 필수 요소이다.

  • $ git remote add <NAME> <GITHUB_REPO_URL>

  • $ git remote add origin https://github.com/howsmyanimeprofilepicture/myrepo.git

    • origin이라는 이름으로 myrepo.git 레포지토리를 추가하겠다는 의미이다.

  • $ git remote get-url origin

    • 위 명령어를 치면 origin이라는 이름으로 등록된 레포지토리의 URL을 가져올 수 있다.

  • $ git push origin master

    • origin이라는 이름으로 등록된 원격 Repo로 (로컬에 있는)master브랜치를 업로드한다는 뜻이다.
    • Github에서 해당 레포에 들어가서 커밋내역들을 확인하면
    • 로컬에서 $ git log를 친 것과 동일하다는 것을 알 수 있다.
    • master브랜치로 관리되는 commit 내역들이 깃헙 Repo로 업로드 된다고 이해하면 좋을 것 같다.

  • $ git pull origin master

 

 

5. commit 되돌리기

5.1. git reset

  • $ git reset

    • $ git reset HEAD~1
    • HEAD란 현재 커밋의 위치를 의미한다.
    • 위 명령어는 HEAD에서 한 커밋 뒤로 되돌리라는 이야기다.
    • --soft --hard --mixed 이렇게 3가지 옵션이 있다. (옵션 안 줄 경우 --mixed가 디폴트 적용)
    • --soft는 Staged 상태로, --hard는 Unmodified 상태로, --mixed는 Modified 상태로 되돌린다.

 

 

5.2. soft, hard, mixed 옵션의 이해

  • 예를 들어 file.txt라는 파일이 있다고 가정하자

    • 📂file.txt

      xxxxxxxxxx
      첫번째 커밋;

      이 상태에서 첫번째 커밋을 하고

    • 📂file.txt

      xxxxxxxxxx
      첫번째 커밋;
      두번째 커밋;

      다음 문장을 추가해서 두번째 커밋을 하고

    • 📂file.txt

      xxxxxxxxxx
      첫번째 커밋;
      두번째 커밋;
      세번째 커밋;

      또 한 문장을 추가해서 세번째 커밋을 했다고 가정하자.

    • $ git reset HEAD~2라는 명령어를 통해 HEAD로 부터 두 커밋 뒤로 되돌려보자.

--hard

  • 이때 $ git reset HEAD~2 --hard를 입력하면 (soft옵션을 사용하면), file.txt에서 두번째 세번째 줄 문장이 사라진 것을 확인할 수 있다.
  • 즉 되돌아간 시점으로 파일까지 변경됨을 확인할 수 있다.
  • Unmodified로 되돌아 간다는 것은 파일이 수정(Modified)되기 전으로 돌아감을 의미한다.
  • 즉, 첫번째 커밋으로 파일의 내용까지 돌아간다.

 

--soft

  • $ git reset HEAD~2 --soft를 입력하면, file.txt는 그대로이다. 두번째 세번째 문장이 여전히 살아있다.
  • $ git status를 찍어보면, stage에 변경된 내용(두번째 세번째 문장이 추가됨)이 올라가있음을 확인할 수 있다.
  • 즉 commit은 첫번째 커밋으로 돌렸지만, 파일들은 그대로이며, 첫번째 커밋시점과 현재 파일들의 차이라고 할 수 있는 부분들은 전부 stage에 올라왔다.
  • 즉 Staged 상태로 돌아갔음을 의미한다.

 

--mixed

  • $ git reset HEAD~2 --mixed를 입력해도, file.txt는 그대로다.

  • 근데 $ git status를 찍어보면, soft와 달리 변경사항들이 stage에 올라와있지는 않다.

  • xxxxxxxxxx
    $ git status
    ​
    On branch master
      (use "git add <file>..." to update what will be committed)  (use "git restore <file>..." to discard changes in working directory)
            modified:   file.txt

    다음과 같은 메시지가 뜬다. 즉 file.txt가 HEAD시점과 다른데 stage에 올라와있지 않다는 소리다.

    (현재는 reset으로 되돌린 관계로, HEAD가 첫번째커밋이 된다.)

    add를 입력해서 변경된 내용을 스테이지에 올리거나, restore를 입력해서 변경사항을 지워 파일의 내용까지 HEAD시점으로 돌리라는 소리다.

5.3. git revert

  • $ git revert HEAD

    • reset과의 차이점은 commit 내역을 지워서 이전으로 돌리는 게 아니라,
    • 커밋을 되돌리는 것을 또 새로운 커밋으로 추가하는 것이다.
    • 예를 들어 $ git revert HEAD라고 입력할 경우, HEAD커밋 바로 이전 커밋으로 복구하는 커밋을 새로 추가된다.
    • 즉 파일 내용은 HEAD~1으로 돌아가지만, 커밋 목록에는 새로운 커밋(revert)이 추가된다.

     

6. branch 관리하기

6.1. git branch

  • $ git branch

    • 만들어진 branch 목록을 가져옴

  • $ git branch NEW_BRANCH

    • HEAD에 NEW_BRANCH라는 새로운 브랜치를 생성한다.

  • git checkout NEW_BRANCH

    • switch와 동일하다. switch가 보다 직관적이고 비교적 최근에 나온 거라 나는 switch를 선호한다.
    • NEW_BRANCH로 이동한다.

  • git switch NEW_BRANCH

  • git switch -c NEW_BRANCH

    • 현 브랜치의 HEAD에 NEW_BRANCH를 생성하고, NEW_BRANCH로 이동한다.
    • git branch NEW_BRANCH; git switch NEW_BRANCH; 를 축약한 것으로 이해하자.
    • git switch -c NEW_BRANCH 143c123c처럼 특정 커밋 시점에 브랜치를 생성하고 해당 브랜치로 이동하는 것도 가능하다.

 

 

6.2. git merge

  • $ git switch master; git merge NEW_BRANCH

    • master브랜치로 다시 돌아가서 방금 생성한 NEW_BRANCH를 병합한다.

    • 만약 master의 HEAD와 NEW_BRANCH의 HEAD가 달라 서로 충돌이 일어난다면

      • $ git merge --abort를 입력해서 브랜치 병합작업을 중단할 수도 있고,
      • 충돌이 일어나는 파일에 접근해서, 원하는 방향으로 수정한 후, $ git add <file>... ; git commit을 통해 브랜치를 병합하는 방법이 있다.

 

 

6.3. git rebase

  • $ git rebase NEW_BRANCH

    • xxxxxxxxxx
      Resolve all conflicts manually, mark them as resolved with 
      "git add/rm <conflicted_files>", then run "git rebase --continue".
      You can instead skip this commit: run "git rebase --skip".
      To abort and get back to the state before "git rebase", run "git rebase --abort".

    • master브랜치의 HEAD와 NEW_BRANCH의 HEAD가 달라서 충돌이 발생하면 위와 같은 메시지가 나온다.

    • 충돌이 발생한 파일을 수정한 후 $ git add <file> ... 를 통해 stage하거나

    • 아니면 해당 파일을 $ git rm <file> ...해서 unstage한 후

    • $ git rebase --continue를 통해 rebase를 계속 진행하라는 소리다.

    • 리베이스를 중단하고자 한다면 $ git rebase --abort를 실행하면 된다.
