1. 별칭 (Type Alias)

같은 필드를 쿼리하지만 해당 필드에 전달되는 인자 값이 다를 경우, Type Alias 사용해야 한다.

{
  empireHero: hero(episode: EMPIRE) {
    name
  }
  jediHero: hero(episode: JEDI) {
    name
  }
}

 

hero 필드를 쿼리하지만 episode 인자에 전달되는 값이 서로 다릅니다. 만약 타입 별칭을 사용하지 않는다면 hero필드는 서로 충돌하게 됩니다. 

xxxxxxxxxx
{
  "data": {
    "empireHero": {
      "name": "Luke Skywalker"
    },
    "jediHero": {
      "name": "R2-D2"
    }
  }
}

(↑ 타입 별칭으로 리턴 받은 데이터)

 

 

2. 프래그먼트

  • Fragment는 재사용 가능한 단위(모듈화 가능한 단위)

  • 필드가 반복될 경우, 이를 작은 단위로 분할하기 위해 사용된다.

  • ​x
    query HeroComparison($first: Int = 3) {
      leftComparison: hero(episode: EMPIRE) {
        ...comparisonFields
      }
      rightComparison: hero(episode: JEDI) {
        ...comparisonFields
      }
    }
    ​
    fragment comparisonFields on Character {
      name
      friendsConnection(first: $first) {
        totalCount
        edges {
          node {
            name
          }
        }
      }
    }

  • 위 예시는 Fragment를 활용해서 { name, appearsIn, Friends { name } } 필드를 모듈화했다.

     

 

 

3. Union Type, 그리고 그에 대한 Resolver

 

  • Union Type은 Type Script의 유니온과 100% 똑같다.

  • Object, Query, Mutation과 마찬가지로 Union도 Resolver를 가진다.

  • 근데 Union은 자신만의 고유한 필드를 갖는 타입이 아니다. 

  • xxxxxxxxxx
    {   
        Given : {
            __resolveType(given, context, info){
                if (given.used_by) {
                    return 'Equipment';
                } // given에 used_by필드가 있다면, Equipment타입으로 결정한다는 뜻
                if (given.team) {
                    return 'Supply';
                }
                return null;
            },
        },
    };

    위 예제는 Given이라는 유니언 타입에 대한 Resolver이다.

    (union Given = Equipment | Supply)

  • Given을 반환하는 Query(혹은 Mutation), 혹은 Given 타입의 필드로 갖는 Object Type에 대한 요청이 들어오면, Given에 대한 Resolver가 실행된다. __resolveType(given, context, info)에 given인자는 이 때 Given을 반환하는 상위 Resolver의 리턴값이라고 이해하면 좋을 것 같다.

  • Given에 대한 Resolver가 Supply 타입으로 판단한다면, Supply 타입의 각 필드에 대한 리졸버가 실행될 것이다.

  • 그리고 이 경우 Given에 대한 Resolver의 given 인자와 결정된 타입의 각 필드의 Resolver의 parent인자는 같은 모양의 객체가 될 것이다.

  • 위 쿼리의 내용을 해석해보자. givens는 [Given]을 반환하는 쿼리이다.

  • __typename은 반환되는 Given이 무슨 Type인지를 나타낸다. Supply 혹은 Equipment가 될 것이다.

  • ... on Equipment는 반환된 Given이 Equipment일 때 응답받고자 하는 데이터(필드)들이다.

  • 반대로 ...on Supply는 ... 뭐 대충 알겠지?

 

 

4. Interface

  • 추상 타입이다. (타입스크립트와 다른 점)

  • 즉, 다른 타입에 implemented 되기 위한 타입이란 소리다.

  • 즉 Interface라는 Type을 갖는 Object(혹은 Instance)는 없다.

  • 인터페이스란 결국 좀 더 큰 체계를 뜻한다.

  • 기본적으로 객체지향의 상위 클래스를 상속받는 것과 유사하다. 추상 타입이라는 것을 기억하자

     

     

    xxxxxxxxxx
    type Equipment implements Tool {
        id: ID!
        used_by: Role!
        count: Int
        new_or_used: NewOrUsed!
    }
    xxxxxxxxxx
    type Software implements Tool {
        id: ID!
        used_by: Role!
        developed_by: String!
        description: String
    }
    xxxxxxxxxx
      const { gql } = require('apollo-server')
      const typeDefs = gql`
          interface Tool {
              id: ID!
              used_by: Role!
          }
      `
      const resolvers = {
          Tool: {
              __resolveType(tool, context, info) {
                  if (tool.developed_by) {
                      return 'Software'
                  }
                  if (tool.new_or_used) {
                      return 'Equipment'
                  }
                  return null
              }
          }
      }
      module.exports = {
          typeDefs: typeDefs,
          resolvers: resolvers
      }

  • 인터페이스 타입을 반환한다는 것은 어떤 의미에서 유니언 타입을 반환하는 것과 매우 유사하다.

  • 인터페이스 단독으로는 어떠한 인스턴스(혹은 데이터)가 될 수 없기 때문이다.

  • 앞서 추상타입이라고 한 것처럼 어떠한 인터페이스를 반환 타입으로 갖는 필드라면 결국 해당 인터페이스를 implemented한 Object Type을 필드로 갖게 된다는 뜻이다.

  • 위의 예시를 참고하면, 만약 어떤 필드의 타입이 Tool 이라는 인터페이스라면, 해당 필드은 Tool이 이식된 Object Type들(위 예시에선 Software와 Equipment)을 타입으로 갖는다는 뜻이다.

  • 다르게 얘기하면 인터페이스 타입은 인터페이스가 이식된 타입들의 유니언이라고 봐도 무방하다.

  • 결국 인터페이스는 상위 타입으로 이해해도 무방하다. 자식클래스도 결국 부모클래스의 한 부류로 볼 수 있으니까말이다.

  • 위의 resolver를 보자. 유니언타입의 resolver와 동일하다.

 

5. 인라인 프래그먼트

다른 여러 타입 시스템과 마찬가지로 GraphQL 스키마에는 인터페이스와 유니온 타입을 정의하는 기능이 포함되어 있습니다. 스키마 가이드에서 대해 자세히 알아보세요.

  • 인라인 프래그먼트는 인터페이스나 유니언 타입을 반환하는 필드를 쿼리하는 경우 사용된다.
xxxxxxxxxx
query HeroForEpisode($ep: Episode!) {
  hero(episode: $ep) {
    name
    ... on Droid {
      primaryFunction
    }
    ... on Human {
      height
    }
  }
}
​
{  variables : {"ep": "JEDI"} }
​
​
{
  "data": {
    "hero": {
      "name": "R2-D2",
      "primaryFunction": "Astromech"
    }
  }
}
  • 이 쿼리에서 hero 필드는 Character 를 반환하는데,
  • episode 인자에 따라서 Human이나 Droid 중 하나일 수 있습니다.
  • 필드를 직접 선택할 때에는 name 과 같이 Character 인터페이스에 존재하는 필드만 요청할 수 있습니다.
  • 특정한 타입의 필드를 요청하려면 타입 조건과 함께 인라인 프래그먼트 을 사용해야합니다.
  • 첫 번째 프래그먼트는 ... on Droid 라는 레이블이 붙어있기 때문에 primaryFunction 필드는 hero 에서 반환된 Character 가 Droid 타입인 경우에만 실행됩니다.
  • Human 타입의 height 필드도 마찬가지입니다.

 

6. 메타 필드

GraphQL 서비스에서 리턴될 타입을 모르는 상황이 발생하면 클라이언트에서 해당 데이터를 처리하는 방법을 결정할 방법이 필요합니다. GraphQL을 사용하면 쿼리의 어느 지점에서나 메타 필드인 __typename 을 요청하여 그 시점에서 객체 타입의 이름을 얻을 수 있습니다.

xxxxxxxxxx
{
  search(text: "an") {
    __typename
    ... on Human {
      name
    }
    ... on Droid {
      name
    }
    ... on Starship {
      name
    }
  }
}
{
  "data": {
    "search": [
      {
        "__typename": "Human",
        "name": "Han Solo"
      },
      {
        "__typename": "Human",
        "name": "Leia Organa"
      },
      {
        "__typename": "Starship",
        "name": "TIE Advanced x1"
      }
    ]
  }
}

위 쿼리에서 search 는 3 가지 중 하나인 유니언 타입을 반환합니다. __typename 필드가 없으면 클라이언트가 다른 타입을 구별하는 것은 불가능할 것입니다.

GraphQL 서비스는 몇 가지 메타 필드를 제공하며, 나머지는 Introspection 시스템을 노출하는 데 사용됩니다.
