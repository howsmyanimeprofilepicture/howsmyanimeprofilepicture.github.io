1. createAsyncThunk의 주요개념

createAsyncThunk 함수는 Thunk를 반환합니다.

Thunk는 함수 형태의 action을 리턴하는 함수를 의미합니다.

일반적인 액션생성함수로 생성한 객체 형태의 action의 경우, action 그 자체에 비동기적인 로직을 담을 수는 없습니다.

반면 Thunk로 생성한 함수로 된 action에는 비동기적인 로직을 담을 수 있습니다.

 

redux-toolkit은 기본값 미들웨어로 redux-thunk를 지원합니다. redux-thunk는 함수 형태의 action을 dispatch할 수 있게 해주는 미들웨어입니다.

thunk에 대한 추가적인 공부는 https://react.vlpt.us/redux-middleware/04-redux-thunk.html

 

아래 용어들을 명확하게 기억하면 이해에 큰 도움이 됩니다.

createAsyncThunk

Thunk

Payload Creator, 페이로드생성자

Promise Lifecycle Action Type

 

 

2. 기본적인 흐름

​x
import { createAsyncThunk, createSlice } from '@reduxjs/toolkit'
import { userAPI } from './userAPI'
​
// First, create the thunk
const fetchUserById = createAsyncThunk(
  'users/fetchByIdStatus', 
    // 👀첫번째 인자로는 action type string이 들어갑니다.
  async (userId, thunkAPI) => {
    const response = await userAPI.fetchById(userId)
    return response.data
  } // 👀두번째 인자로는 콜백함수가 전달되는데요. 
    // 👀이 콜백함수를 Payload Creator라고 합니다.
    // 👀페이로드 크리에이터가 반환하는 값이
    // 👀최종적으로 action.payload가 됩니다.
)
​
// Then, handle actions in your reducers:
const usersSlice = createSlice({
  name: 'users',
  initialState: { entities: [], loading: 'idle' },
  reducers: {
    // standard reducer logic, with auto-generated action types per reducer
  },
  extraReducers: {
    // Add reducers for additional action types here, and handle loading state as needed
    [fetchUserById.fulfilled]: (state, action) => {
      // Add user to the state array
      state.entities.push(action.payload)
        
    }
  }
})
​
// Later, dispatch the thunk as needed in the app
dispatch(fetchUserById(123))
​

 

위 예제를 참고해볼까요?

상수(const) fetchUserById에 createAsyncThunk가 반환하는 함수가 담깁니다.

즉, fetchUserById가 바로 Thunk입니다.

dispatch( fetchUserById(123) ); 과 같은 방식으로 thunk가 반환하는 action을 디스패치 할 수 있습니다.

 

 

우선 createAsyncThunk에 대해 더 알아봅시다.

첫번째 인자는 액션 타입입니다.

중요한 건 두번째 인자로 전달되는 payload creator(페이로드 생성자)입니다.

페이로드 생성자의 첫번째 인자는 Thunk가 호출될 때 Thunk에 전달되는 인자입니다.

위 예시에서 dispatch( fetchUserById(123) );를 실행하면, fetchUserById에 전달된 123이 바로

payload creator의 첫번째 인자값이 되는 것입니다. 원리가 이해가 되실까요?

 

 

3. Thunk가 반환하는 action을 디스패치하면?

함수를 디스패치한다는 것의 의미를 한 번 생각해야 합니다.

함수 그 자체에는 그 어떠한 type프로퍼티도 존재하지 않습니다.

함수를 디스패치하는 것, 다시 말해 thunk를 디스패치하는 것은 사실 어떠한 action도 발생시키지 않는다는 의미입니다.

thunk가 반환하는 함수(함수형태의 액션)의 내부에는

여러 비동기적 로직과 dispatch함수가 들어가 있습니다.

즉 thunk의 디스패치는 액션을 발생키는 것보다는

여러 액션 디스패칭의 비동기적 로직을 발생시킨 다는 것이 더 맞습니다.

 

위 예시를 기준으로 설명하면,

만약 dispatch(fetchUserById(123)); 과 같이 thunk가 디스패치되면 아래와 같은 비동기적인 로직이 실행됩니다.

  1. { type : "users/fetchByIdStatus/pending" }을 디스패치합니다.
  2. payloadCreator를 호출합니다. 그리고 프로미스가 resolve되기를 기다립니다.
  3. 프로미스가 잘 이행되면, { type : "users/fetchByIdStatus/fulfilled" , payload : payloadCreator(프로미스)의 결과값 }이 디스패치 됩니다.
  4. { type : "users/fetchByIdStatus/rejected"}가 디스패치 됩니다.

 

 

 

rejectWithValue(value)를 통한 error핸들링이 있는 경우에는 payloadCreator의 프로미스가 rejected된 경우에도 payload 프로퍼티가 생기지만... 너무 많습니다....
