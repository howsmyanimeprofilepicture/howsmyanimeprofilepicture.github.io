1. Provider.of 메서드를 사용할 경우의 리빌드 범위

xxxxxxxxxx
Provider.of<CountProvider>(parentBuilderCtx, listen: true);
  1. 위의 코드는 parentBuilderCtx라는 빌드컨텍스트 객체를 기준으로 위젯트리상 위에서 제일 가까운 CountProvider 객체를 참조한다.
  2. listen이 true일 경우가 중요한데, parentBuilderCtx 기준으로 위젯트리상 아래에 위치한 모든 위젯은 CountProvider 객체가 관리하는 state가 변할 때마다 리빌드된다.
  3. 리빌드 되는 범위는 Provider.of 메서드가 호출된 곳이 어디인가가 따라 결정되는 것이 아니다.
  4. 인자로 전달된 빌드컨텍스트가 무엇이냐에 따라 결정된다.
  5. 위의 예시를 보면 Provider.of()에 parentBuilderCtx를 전달하면, '부모 빌더 위젯 빌드콜' => '빌더 위젯 빌드콜' => '텍스트 최하단 빌드콜'이 순서대로 찍히게 된다.
  6. 만약 Provider.of()에 builderCtx를 전달했다면, state가 변경될 때마다 '빌더 위젯 빌드콜' => '텍스트 최하단 빌드콜'까지만 찍힐 것이다. parentBuilderCtx는 builderCtx보다 상위에 있는 컨텍스트이기 때문에 '부모 빌더 위젯 빌드콜'은 생략된다. 즉 parentBuilderCtx를 갖고 있는 위젯(가칭 : 부모 빌더 위젯)은 리빌드 되지 않는다는 소리다.

 

2. Cousumer 위젯의 리빌드 범위

 

 

  1. 전에도 설명했듯 Consumer 위젯의 경우, 구독하는 프로바이더의 스테이트 변화에 따라, Cousumer자신과 자손 위젯들이 리빌드 된다.
  2. 위의 그림과 같은 트리구조를 지니고 있다면 CountProvider의 State가 변할 떄
  3. Consumer<CountProvider>위젯과 그 자손위젯(Text)가 리빌드 된다.

import 'package:flutter/cupertino.dart';
import 'package:flutter/material.dart';
​
class CountProvider extends ChangeNotifier {
  int _count = 0;
​
  int get count {
    return _count;
  }
​
  void add() {
    _count++;
    notifyListeners();
  }
​
  void remove() {
    _count--;
    notifyListeners();
  }
}
